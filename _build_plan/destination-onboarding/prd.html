Destination Onboarding

Rapture should land captures where the user's notes actually live — on day one, not a week later. The app detects the notes vaults already installed on the Mac and offers them as the destination, contained in a single subfolder so its output never scatters among the vault's own folders; it nudges existing users still sitting on the silent default; and it asks before moving any notes, instead of relocating them invisibly.

Swift 5.9+ SwiftUI macOS 14+ MVVM · @Observable XCTest Unsandboxed 3 milestones

Why this exists. The triage engine (v1.0.98) already does the hard part: every capture becomes a classified Markdown note the moment it arrives. This feature is the last mile — the gap between "perfectly triaged" and "actually in my vault."

That gap was measured in a real dogfood. The author wanted captures in an Obsidian vault, was never prompted to repoint, and a week of correctly-triaged notes piled up in ~/Documents/Rapture Notes — stranded outside the vault and synced nowhere. He then hand-moved them, unaware the app has shipped automatic, ledger-aware relocation since v1.0.69 — and the hand-move desynced both ledgers (4 of 4 triage records and 2 of 2 enrichment records pointed at files that no longer existed). Every failure in that story is a discoverability failure, not a missing capability.

What the app does

  • Finds your real notes vault

    Reads Obsidian's own config and offers each vault by name, instead of defaulting silently to a folder you never chose.

  • Suggests sync roots too

    iCloud Drive, Dropbox, and Google Drive appear as secondary options — a synced destination is what makes notes reachable from the phone.

  • Asks on first run

    After Full Disk Access is granted, you choose where notes go — the one moment where asking prevents a stranded week.

  • Never loses a capture while you decide

    The default folder stays underneath as a safety net, so a note dictated before you answer lands safely and migrates when you pick.

  • Keeps its output contained

    Point at a populated vault and it offers one container (default Rapture Inbox) instead of scattering five folders among your own.

  • Asks before it moves anything

    Shows how many notes move, from where to where, and how many collide — then lets you move them, leave them, or cancel.

  • Nudges you off the default

    A dismissible notice when a vault is detected and you're still on the default. It never switches anything on its own.

  • Rescues a scattered vault root

    If Rapture's folders are already loose in a vault root, it offers to gather them into a container.

Already provided by the existing codebase

This feature extends shipped machinery. None of this is re-specced or rebuilt.

  • OutputFolderMigrator — relocation engine (v1.0.69)
  • AppState.setOutputFolder — orchestration + CaptureGate quiesce
  • TriageLedger / EnrichedLinkLedger + remap
  • DestinationGuard + offline spool
  • OutputFolderSidecar — the public path contract
  • OutputFolderScaffold — opt-in CLAUDE.md seed
  • HandoffEnableFlow / AITriageEnableFlow — the consent pattern
  • triageIntroShown — the dismissible notice pattern
  • PermissionsView — the FDA onboarding, stays as step one

Out of scope

  • Rebuilding migration — OutputFolderMigrator ships and works. The gap is consent and discoverability, not capability.
  • Security-scoped bookmarks — the app is unsandboxed; zero usages in the codebase and tech-stack.md says explicitly none is needed.
  • Note-level dedup for re-dictations — reported as a bug, but files-always-land is the mission.md commitment. Handoff dedup is separate and verified working.
  • Apple Notes, Bear, Notion as destinations — database-backed, no folder to point at. Filing into them needs an importer, which breaks output neutrality.
  • Obsidian-flavored output — mission.md's hard commitment: output must never be readable only by one vendor's tool. Detection decides where to write, never how.
  • Multi-folder or per-class destinations — mission.md commits to one output folder per install.
  • Two-way sync — Rapture writes notes; it never reads the vault back or reacts to edits.
  • Backfill / re-triage after moving — bulk re-triage is out per mission.md; enrichment set the no-backfill precedent.
  • Following a vault that moves — relocate it in Finder and you repoint. No filesystem surveillance.
  • Touching the vault's existing content — Rapture never renames, reorganizes, or reads the user's own notes.
  • Installing Obsidian for you — no vault found means the default folder is the answer.
  • Cloud/remote destinations (S3, WebDAV) — local paths only; sync engines handle the rest.
  • Vault git auto-backup — considered and deliberately rejected. A git push is a subprocess, so PRIVACY's grep verification claim would silently go stale; hourly add -A would race the AI sessions that commit to that vault. Solved by a signed helper app outside this codebase.

External integrations

None. No providers, no credentials, no API keys, nothing to sign up for.

Vault detection reads one local JSON file that Obsidian already maintains. This matters beyond convenience: this feature adds no outbound network path. The app's three enumerated paths in PRIVACY.md (Sparkle updates, the BYO-key Anthropic engine, the link-enrichment fetcher) stay exactly three, and PRIVACY's grep claim still returns the same three files. Any implementation that would change that is out of scope by definition.

Data model

What the app needs to remember. The destination is already modelled (outputFolder in settings.json) and containment is just a longer path — so this feature adds almost nothing.

PersistedState — one new field

defaultDestinationNudgeDismissed Whether the user has settled the question of the default folder. Set when they either pick "keep the default" at first run or dismiss the nudge. One flag covers both, so someone who deliberately chose the default is never nagged. Defaults to no; must decode leniently — a strict key would wipe every existing user's ledgers.

Lives in · state.json

Detected vault

name The vault's display name, as the user knows it (e.g. "Second Brain").
path Where it lives on disk.
source How it was found — an Obsidian vault, or a sync root like iCloud/Dropbox.
reachable Whether its drive is connected right now.

Never persisted · re-read each time the picker opens, so a new vault appears without a stale cache and an unplugged drive shows as unreachable immediately.

Milestones

1

Vault-Aware Destination

Teaches the existing "change your notes folder" path to know about vaults, contain its output, and ask before moving. Delivered entirely through Settings → General, which already exists. This milestone alone would have prevented the stranded-week problem.

What gets built

  • Picker offers detected vaults by name, plus sync roots as secondary suggestions
  • A vault on a disconnected drive is shown but not selectable, with the reason stated
  • Populated folder → offer to contain under an editable name, default Rapture Inbox
  • Existing container name: adopt when empty or already ours; otherwise suggest another
  • Consent step showing count, source, destination, and collision renames
  • Move them / Leave them behind (prunes the ledger) / Cancel
  • Sidecar always reflects the final destination, container included

Not in this milestone

  • Any change to first launch — the default is still created silently
  • The default-folder nudge or any proactive notice
  • The vault-root rescue
  • Any change to how notes are named, formatted, or classified
  • Any new network call, credential, or entitlement

Done when: You can open Settings → General, see your real Obsidian vault offered by name, pick it, be offered Rapture Inbox as a container, see exactly what's about to move, and decline or accept — with notes ending up where you chose, the ledgers still resolving, and the sidecar pointing at the container. Unplugging the drive shows the vault as unreachable rather than gone.

2

First-Run Destination Flow

Replaces the silent default with an actual question, in the one place where asking prevents a week of stranded notes.

What gets built

  • After FDA is granted, the app asks where notes should go
  • Detected vaults, "Choose another folder…", and an explicit "Keep the default"
  • Picking a vault runs M1's containment offer and migration consent
  • "Keep the default" is respected permanently — no nudge ever appears
  • The default folder stays underneath as a safety net — no capture waits on a decision
  • Dismissing without answering leaves the question open for the nudge

Not in this milestone

  • Changing, restyling, or reordering the Full Disk Access step
  • Blocking capture on the destination choice
  • The nudge or the vault-root rescue
  • Any onboarding beyond the destination question — no tour, no feature intro

Done when: A clean install walks from launch → FDA grant → destination choice → notes landing in the chosen vault, contained. Dictating a note before answering still files safely to the default and moves when the choice is made.

3

Nudges & Vault-Root Rescue

Reaches the users milestones 1 and 2 can't: everyone who already installed Rapture and is sitting on a silent default, or scattered across a vault root.

What gets built

  • Dismissible menu-bar notice naming the detected vault, offering to move
  • A quiet, permanent Settings line that survives dismissal
  • Both route into M1's containment and consent — neither switches on its own
  • No notice when no vault is detected, or once the question is settled
  • Vault-root rescue: gather loose Rapture folders into a container
  • The rescue works despite the migrator refusing nested paths

Not in this milestone

  • Repeated or recurring nagging — dismissed stays dismissed
  • Any notice when no vault is detected
  • Auto-switching, auto-containing, or auto-moving under any condition
  • Rescuing anything other than Rapture's own folders

Done when: An existing install on the default folder, with a vault present, shows the notice on next launch; dismissing hides it forever while the Settings line remains. A destination set to a vault root with scattered Rapture folders offers the rescue, and accepting gathers them into Rapture Inbox with the ledgers still resolving.
