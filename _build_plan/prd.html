Rapture for Mac

A menu-bar app that turns Siri-dictated iMessages into timestamped .txt files in a folder any AI assistant can read. Frictionless capture from a locked iPhone — no tap, no button, no shortcut to install — with an in-thread ✓ Saved reply confirming each capture from across the room.

Swift 5.9+ SwiftUI · MenuBarExtra macOS 14+ (Sonoma) GRDB.swift No sandbox Apache-2.0 Signed DMG · GitHub Releases

What the app does

  • Live iMessage capture

    Polls chat.db once per second; decoded text + attachments land in the user's folder within ~1s.

  • Self-chat + allowlist

    Self-chat always captured (no setup). User-managed allowlist for other contacts. SMS off by default — sender IDs are spoofable.

  • In-thread ✓ Saved reply

    iPhone's standard chime is the audible confirmation. Mode picker: All / Errors only / Off.

  • Catch-up on launch

    Watermarked replay of missed messages after wake/quit. 4+ messages collapse into one summary reply.

  • Menu-bar status + controls

    Today's count, last capture time, last error, pause/resume, open folder, quit. No Dock icon.

  • Settings window

    Three tabs — General (folder, reply mode, launch-at-login), Allowlist (handles editor), About (version, diagnostics).

  • Permissions onboarding

    Modal sheets with deep-links to System Settings for FDA and Automation → Messages, with plain-language explainers.

Already provided by the existing codebase

Nothing yet. This is a fresh Xcode scaffold — no pre-built auth, app shell, settings system, or persistence layer to omit. Everything from Milestone 1 onward is yet to be built. The closest thing to a "starter" is Apple's default Xcode macOS app template (an empty App struct plus a MenuBarExtra scene), which provides essentially nothing of substance for this app.

Out of scope (v1)

  • Cloud mode (Sendblue) — deferred to v1.1 with a VPS-relay architecture; an on-Mac webhook would die when the Mac sleeps.
  • Group chat support — chat_style == 43 filtered out at the source.
  • Contacts framework name resolution — allowlist uses raw handles only, no friendly-name lookup.
  • Edit / unsend tracking — captures are point-in-time; later edits don't propagate.
  • Mac App Store distribution — structurally impossible (sandbox blocks chat.db and the osascript subprocess used for replies).
  • Auto-update (Sparkle) — manual download from GitHub Releases for v1.
  • Analytics — no PostHog, TelemetryDeck, or telemetry of any kind.
  • Routing Rapture iOS dictations into this folder — sibling-repo concern (claude-channel-rapture / rapture-api-gateway); would break the spec's "no networking in v1" boundary.
  • In-app browsing / search / preview — the folder is the UI; use Finder, Spotlight, ripgrep, or your AI assistant.
  • In-app editing / tagging / categorizing — .txt files are immutable; downstream AI decides what to do with them.
  • Built-in AI / LLM integration — capture layer, not the AI. Vendor neutrality is the differentiator.
  • Audio capture of the original dictation — text only; audio lives in iMessages on the iPhone.
  • Multi-folder destinations — one output folder per install.
  • Mac Notification Center pings per capture — the iMessage reply is the confirmation; a Mac notification would be redundant noise.
  • Encryption / password protection of captures — point the output folder at an encrypted volume if needed.

Data model

What the app needs to remember. Two small JSON files in ~/Library/Application Support/Rapture for Mac/, atomic .tmp → rename(2) writes.

Settings

outputFolderThe folder the user picked for captures (security-scoped bookmark, survives across launches).
allowedHandlesPhone numbers / Apple IDs whose messages should be captured beyond self-chat.
allowSMSWhether to capture SMS/RCS too. Default off — sender IDs are spoofable.
launchAtLoginAuto-start with the Mac.
pausedWhether capture is currently paused (set from the menu bar).
replyModeAll / ErrorsOnly / Off.

File · settings.json

PersistedState

chatDbWatermarkHighest ROWID processed; catch-up reads everything above it.
selfHandlesCacheTsWhen the user's own iMessage handles were last refreshed.
recentSentEchoesLast ~15s of replies the app sent; used to drop those if they return as inbound.
lastErrorMost recent capture failure, if any (surfaced in the menu bar).

File · state.json

Output artifacts live in the user's chosen folder, not in the app's state: one <timestamp>.txt per captured message (UTF-8, optional Attachments: footer), and a sibling <timestamp>/ folder per message-with-attachments containing the copied files.

Milestones

1

First Capture

The app captures qualifying iMessages and writes them to disk. No UI, no replies — the app runs invisibly and the user's only evidence is .txt files appearing in ~/Documents/Rapture Notes/.

What gets built

  • Xcode project scaffold (RaptureMac.xcodeproj), macOS 14+ target, LSUIElement=YES, no sandbox, GRDB via SPM
  • Models, atomic JSON stores for Settings and PersistedState
  • AttributedBody decoder (so Siri-dictated messages surface)
  • chat.db watcher (1s polling, ROWID watermark)
  • Self-handle resolver + filter (9 drop rules, self-chat default)
  • File writer (atomic .tmp → rename(2)), attachments to sibling folder
  • Default output folder ~/Documents/Rapture Notes/ auto-created on first launch
  • FDA onboarding sheet + deep-link to System Settings + 2s polling

Not in this milestone

  • Menu-bar UI (popover, status, controls)
  • ✓ Saved reply (Milestone 2)
  • Catch-up on launch (Milestone 2)
  • Settings window or allowlist editor (Milestone 3)
  • Code signing or distribution (Milestone 4)

Done when: The user grants FDA, leaves the app running invisibly, sends themselves an iMessage from any iCloud device, and within ~1s sees a <timestamp>.txt appear in ~/Documents/Rapture Notes/ with exactly the dictated text. Sending with a photo also produces the sibling folder with the copied image.

2

Confirmation & Recovery

Every capture is now visibly confirmed on the user's phone, and nothing gets dropped when the Mac sleeps or quits. Still no UI for the user to interact with — but the trust mechanisms are in place.

What gets built

  • AppleScript sender (osascript subprocess) posting via Messages.app
  • Replier composing ✓ Saved: <filename> on success, ✗ <reason> on failure
  • 15s echo guard so the app's own replies don't re-capture
  • Automation → Messages pre-prompt explainer before the OS dialog
  • Catch-up detection (first batch > 3 → catch-up mode)
  • Per-message replies for 1–3 missed; 📥 Caught up: N (M failed) summary for 4+
  • UNUserNotification fallback plumbing (toggle lands in M3)

Not in this milestone

  • Menu-bar UI (Milestone 3)
  • Settings window or reply-mode picker — hardcoded to .all for now
  • Allowlist editor (Milestone 3)
  • Code signing / distribution (Milestone 4)

Done when: User sends a self-message and receives ✓ Saved: <filename> within ~3s. Quits the app, sends 5 messages from another device, relaunches, sees 5 .txt files appear and one summary reply 📥 Caught up: 5 notes captured. The reply doesn't re-trigger a capture.

3

User Control

The user can now see what the app is doing and change how it behaves — without quitting. Full menu-bar UX plus the Settings window with all three tabs.

What gets built

  • Menu-bar popover: status line, today's count, last capture time, last error, pause/resume, open folder, settings, quit
  • Menu-bar icon visually changes when paused
  • Settings → General: folder picker (NSOpenPanel + bookmark + drag-and-drop), launch-at-login (SMAppService), reply-mode picker wired up, allow-SMS toggle
  • Settings → Allowlist: add/remove handles (phone or email)
  • Settings → About: version (git-commit-count), repo link, license, diagnostics, acknowledgments
  • Empty-allowlist hardcode from M1 replaced with the real editable list

Not in this milestone

  • Code signing or notarization (Milestone 4)
  • DMG packaging (Milestone 4)
  • Public GitHub repo flip (Milestone 4)
  • Auto-update, analytics (out of scope for v1)

Done when: User clicks the menu-bar icon, sees ✓ capturing · Today: 7 · Last: 2 min ago, pauses and resumes capture (with visible icon change), opens Settings, picks a different folder, adds an allowlisted contact, switches reply mode to "Errors only", and confirms the new behavior matches.
