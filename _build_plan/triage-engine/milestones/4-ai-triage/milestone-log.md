## What was built

**Where AI sits (the load-bearing shape):** the AI call happens **inside the four composers, just before compose** — the only places that hold the capture text and decide type/title/body — so classification, smart title, formatted body, and `## Raw` all land in the note's single atomic write. Voice-notes only (`TriageClassifier` runs first; link captures never consult AI — M5 enrichment keys off `youtube-link`/`article-link`), `.full` mode only, spool-enqueue never (AI runs at flush, against the item's own `capturedAt`). The result rides back to the processors via `WriteResult.ai` so `HandoffManager` gets the AI's handoff candidates without a second read. Relay-derived iPhone titles still beat AI titles (provenance wins).

## Decisions not pre-specified in the PRD

1. **Settings layout: a dedicated "Triage" tab** (user decision) — filing mode + AI + Reminders/Calendar handoff consolidated; General keeps folder/launch/reply/SMS/iPhone. M5's enrichment toggle joins this tab.
2. **Anthropic model: `claude-haiku-4-5`** (user decision) — fastest/cheapest tier, squarely sufficient for 4-way classification of short dictations; it's the user's own key paying per capture.
3. **AI inside the composers, not the processors** (plan-agent refinement) — three of the four seams only hold the text inside their writer; `WriteResult.ai` echoes the result out. Zero writer-protocol churn, all pre-M4 tests compiled untouched.
4. **Timeout 10s, cooldown 2 strikes → 60s, 401 latch.** Haiku answers in ~1–3s and on-device in a few; 10s is the ceiling that still feels like "seconds" for a background filer, and the cooldown bounds a dead network's cost across a 50-note drain to ~20s total.
5. **Handoff merge: AI replaces the detector — except when its handoff block was entirely discarded by validation** (`handoffsInvalidated` → deterministic detector), so a hallucinating model can't silently disable the M3 behavior the user already trusts. A validated *empty* AI list IS trusted (superset assumption). The manager's shared gates (toggles, auth status-only, past-event skip, notes-field contract, reply suffix) apply to AI candidates unchanged; 1+1 cap kept.
6. **The M3-flagged dedup gap closed with dual fingerprints:** every creation records and checks BOTH the mechanical title fingerprint (M3 format, unchanged) and a new clause fingerprint (`kind|clause|normalizedTitle(clause)|dateKey`). The same re-dictated utterance blocks deterministic↔AI double-creates in either direction, even though AI smart titles drift. Pre-M4 title-only ledger entries are an accepted one-time seam; 2 entries/creation halves effective capacity to ~250 items (cap 500 unchanged) — fine at this scale.
7. **Toggle-enable refuses to persist ON with no engine** (resolution check, honest caption pointing at the key field). If an engine disappears later the toggle stays on, the status line says why, filing continues deterministically.
8. **Formatted-body conservatism:** discarded whenever input was clipped (never "format" text the model didn't see), equal to raw (no redundant `## Raw`), or outside ±50% of raw length (that's a rewrite, not light formatting). `rawBody` is set only when a formatted body is actually used — the M1 `## Raw` compose invariant needed zero changes.
9. **Prompt design** (manual-iteration surface, single source of truth in `AITriagePrompt.instructions`): classification rules distilled from the user's mined rulebook (tasks = actionable imperatives; ideas = product/content; journal = observations/feelings/gratitude; **null when unsure — misfiling is worse than not classifying**); title examples mirror the rulebook's worked examples; handoffs require explicit triggers or stated date+time, verbatim clause mandatory.
10. **Engine resolution is fresh per capture** (Apple availability and the key can change mid-run; toggle read at call time → flips mid-drain apply to remaining files only; no retroactive re-triage in either direction).
