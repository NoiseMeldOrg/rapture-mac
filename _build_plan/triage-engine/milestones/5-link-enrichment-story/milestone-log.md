## What was built — code half

**New source — `RaptureMac/RaptureMac/Enrichment/`** (folder-sync project, no pbxproj edits):

- `LinkEnrichment.swift` — the seam: `LinkEnriching` protocol (`noteFiled(noteURL:in:echo:)`, synchronous/non-blocking on the main actor so gate-holding processors call it safely), `LinkNoteEcho` (type/rawMedia/capturedAt — echo presence IS eligibility), `FetchedLinkContent`, `LinkFetchError` with the load-bearing `isTransport` split (transport errors retry; `noCaptions`/`unusableContent` give up at once).
- `LinkEnrichmentService.swift` — the worker: in-memory FIFO (no persisted queue, no backfill — the "arrival window" is the job's lifetime), one job at a time, two phases. **Phase A fetch, capture gate NOT held** (20s attempt budget via `AITriageService.withTimeout`; 3 transport attempts spaced 30s/120s, injectable; 60s cooldown after 2 consecutive transport-failed jobs). **Phase B mutation, ONE gate hold**: re-check toggle/pause/relocation/`DestinationGuard`, resolve destination-relative paths against the *current* folder, `fileExists` guards, then rename → `TriageLedger.remap` → artifact write → `Media:` append → `EnrichedLinkLedger.record` (ledger-last: any crash leaves valid notes and at worst an unrecorded artifact a re-capture re-fetches). Paused/relocating/volume-absent defers 10s-cadence up to 5 min, then drops silently. In-flight coalescing: same fingerprint merges onto the queued job — one fetch services every concurrent duplicate; post-mutation stragglers re-enqueue and hit the ledger (zero fetch).
- `LinkFetcher.swift` + `URLSessionLinkFetcher.swift` — the protocol seam and the app's third networking file (PRIVACY names it). `isRunningXCTests` front-guard first line of both entries, 10s per-request timeout, 5 MB response cap, browser UA. Article flow: plain GET, charset from the response with UTF-8 fallback, parse off the main actor.
- `YouTubeTranscript.swift` — pure, golden-tested: brace-balanced `ytInitialPlayerResponse` extraction, captionTracks parse, track preference (manual-en > ASR-en > first), json3 → flowing paragraphs (break on >4s gaps or ~700 chars; timestamps discarded — locked user decision), innertube body (iOS client), oEmbed title parse.

### The mid-verify fix (real-world YouTube drift)

First live run gave up in 1.3s: the watch page's embedded caption `baseUrl`s now return **HTTP 200 with an empty body** (proof-of-origin-token requirement — confirmed with curl and python too, so not a URLSession fingerprint issue), and the innertube **WEB** client returns `UNPLAYABLE` with zero tracks — so the original watch-page-first design found 6 dead tracks, skipped the fallback, and quietly gave up. Live comparison showed the innertube **iOS client** (`clientName: IOS`) returns playable tracks whose URLs serve real json3 (ANDROID works too but serves XML regardless of `fmt`). The fetcher was restructured: innertube-iOS primary, watch page demoted to fallback, and an empty transcript from one source now falls through to the other. This is exactly the fragility the PRD priced in; the quiet-failure posture held (the notes were always complete), and the pure-parser tests needed only the innertube-body golden updated.
